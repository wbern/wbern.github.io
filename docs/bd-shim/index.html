The first measurements.

I had already moved Beads from a remote Dolt host to one local, gc-managed server. That removed the network hop, and made the remaining cost easy to see: a fresh bd process repeatedly rebuilt work the controller already knew.

The July 21 harness compared four paths for the same read-only commands: through gc with and without the shim, and directly through each client. It timed wall time, child CPU, exit status, and a semantic JSON fingerprint.

The turn in the road.

Saren’s “hotspots” instinct helped me look for repeated work rather than a smaller binary.

On 15 interleaved runs, the direct thin client was much quicker for both show --json and filtered list --json. But wrapping those calls in gc still spent most of the trip starting the whole command path.

The same archive recorded two strict-parity failures. That mattered as much as the timing: a fast answer was not yet a replacement.

What survived.

The result was not “replace bd.” It was an opt-in client that can ask the warm controller about one narrowly allowed request. Everything else keeps running real bd, in the caller’s own scope.

The later exact JSON show shortcut is a deliberately small proof. It is faster only where its output contract has been checked, and it falls back everywhere else.

Issue #4441 holds the working discussion. The exploratory harness and raw artifacts are there too.
